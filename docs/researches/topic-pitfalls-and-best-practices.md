# AI Pitfalls & Best Practices for BCE Workshop
## Research Document: Concrete Patterns for 400 Non-AI-Native Mining Professionals

---

## Executive Summary

AI is not a replacement tool—it's an **amplification tool**. The critical difference between teams that harness AI effectively versus those who waste time and resources comes down to **preparation, discipline, and measurement**.

This document catalogs seven major pitfalls that appear repeatedly in production AI systems, maps each to one or more concrete best practices that prevent it, and includes a mindset reframing to address the legitimate fear: "Will AI replace me?"

**Key insight:** Organizations that invest in **upfront planning, clear specifications, and continuous evaluation** ship faster, waste less compute, and build systems that actually work. Those that treat AI as "just throw a prompt at it" build expensive, fragile systems that require constant hand-holding.

## Part 1: Pitfalls Catalog

### Pitfall #1: AI Slop (Low-Quality Generated Content)

**What is it?**
AI slop is high-volume, low-effort output—text, images, or analysis that is generic, hallucinated, or obviously machine-generated. It's the difference between "this sounds like a corporate template" and "this reflects actual industry knowledge."

**Concrete symptom:**
- A geotechnical report generated by AI that says "Coal quality varies by location" with no specific data about Berau deposits.
- Safety procedures that read like Wikipedia summaries, missing site-specific hazards.
- Email templates that work for any industry, not mining.

**How it manifests in production:**
- Teams publish AI-generated content without review → customers recognize it as slop → credibility dies.
- Cost: reputation damage, wasted compute, buried by search engines (Kagi Search now ranks slop lower).

**Real example:** Clarkesworld science fiction magazine shut down submissions in 2024 due to the volume of low-quality AI writing.

---

### Pitfall #2: Hallucination & Bias

**What is it?**
AI models generate false information with high confidence (e.g., invented citations, incorrect facts) and perpetuate biases learned from training data (e.g., gender, cultural, political bias).

**Why it happens:**
Models are prediction engines, not knowledge bases. They predict the statistically likely *next word*, not the *true* next word. When uncertain, they fill gaps with plausible-sounding fabrication. Models trained on internet data learn both truth AND societal biases.

**Concrete symptom:**
- "Coal production in Berau increased 15% in 2024" (sounds reasonable, but the model invented it).
- Safety AI that recommends protective measures differently for different demographic groups.
- Tool recommendations biased toward suppliers the model saw mentioned most (not best-fit).

**Current severity (2026):**
- Frontier models hallucinate 3.1%–19.1% of the time (vs. 15–45% in 2024).
- Citation accuracy is worst-performing: 12.4% hallucination rate even with extended thinking.
- Models are **34% more confident** when making up facts ("definitely," "without doubt").

**How it manifests:**
- Polished-sounding wrong answers that bypass human review.
- Biased recommendations that seem reasonable until deployed.

### Pitfall #3: Token Inefficiency (Bloated Prompts, Repeated Context)

**What is it?**
Wasted tokens = wasted money + slower responses. Giving AI:
- The entire 50-page mining manual when it only needs 3 pages.
- Repeated context across multiple turns (copy-paste the same background info repeatedly).
- Vague prompts that force clarification loops (user asks, AI asks back, user re-explains).

**Concrete symptom:**
- A prompt that loads the entire equipment maintenance handbook into every request.
- Chain-of-thought instructions that newer models already do internally (wasted tokens).
- "Could you please consider summarizing" instead of "Summarize this."

**Cost:**
- Techniques like trimming, compression, and batching can reduce token usage by **40–60%** without losing quality.
- Repetitive context also slows down model responses (attention budget shrinks as context grows).

**How it manifests:**
- Monthly AI costs 3x higher than they should be.
- Response times slow down as prompt history grows.
- Every agent interaction adds more context, making each turn slower.

---

### Pitfall #4: Massive Correction Loops (Agent Stuck Fixing Its Own Mistakes)

**What is it?**
An AI agent loops endlessly:
- Makes a mistake (e.g., calls the wrong tool).
- Tries to fix it (fails again).
- Tries again (still wrong).
- Users wait hours while the system burns tokens.

**Concrete symptom:**
- Agent tries to query a database, gets an error, keeps using the same failed query.
- Agent claims it completed a task but actually never executed it ("the ghost action").
- Agent keeps asking for information the user already provided ("the interrogation loop").
- System racks up $500+ in API costs on a task that should cost $5.

**Real failure modes (2026 production data):**
1. **The ghost action:** Agent claims to delete a file but never did; loop of false fixes.
2. **The interrogation loop:** Agent re-asks for the site name 10 times per conversation.
3. **The confident fabricator:** Agent confidently delivers wrong answers built on bad data.
4. **The budget burner:** Task completes, but wallet is empty.

**How it manifests:**
- Tasks that work in testing fail in production with real, messy data.
- No visibility into what went wrong or how to prevent it next time.
- Cost overruns with no explanation.

### Pitfall #5: Tools Without Direction (Access ≠ Strategy)

**What is it?**
Giving an AI agent access to 20 tools without clear strategy:
- Agent gets confused about which tool to use.
- Multiple tools do overlapping things.
- Agent calls the wrong tool confidently.

**Concrete symptom:**
- Agent has access to both "query_ore_grade" and "fetch_ore_samples"—when should it use which?
- Agent has write access to critical equipment logs but no guardrails.
- Agent can call external APIs, internal databases, *and* email—no priority or safeguards.

**Why it fails:**
The research is clear: "If a human engineer can't definitively say which tool should be used in a given situation, an AI agent can't be expected to do better."

**How it manifests:**
- Agent makes wrong choices under stress or with ambiguous requests.
- Data integrity issues (writes to wrong location).
- Security gaps (agent emails sensitive data).

### Pitfall #6: Building Without Preparation / Strategy ("Prompt-and-Pray")

**What is it?**
Teams skip planning and jump straight to: "Let me prompt an AI to solve this."

No:
- Written spec for what "done" looks like.
- Test cases or success criteria.
- Evaluation plan.
- Fallback if the AI fails.

**Concrete symptom:**
- "I'll use ChatGPT to write the safety procedure" (no spec, no review plan).
- "The AI should handle all shift scheduling" (no definition of "handle," no edge cases planned).
- "Let's use AI for equipment diagnostics" (no test cases, no failure budget).

**Why it fails:**
Without a spec, the AI generates something that *sounds* reasonable but may miss critical details. Without evals, failures reach production.

**Cost:**
- Rework, embarrassment, potential safety issues.

**How it manifests:**
- Output that needs heavy revision.
- Edge cases that break the system.
- No way to measure improvement.

---

### Pitfall #7: Anti-Patterns in Agentic Workflows

**What is it?**
Long-running AI agents (systems that operate independently for hours/days) have unique failure modes:

1. **Over-prompting:** Stuffing 100 rules into the system prompt doesn't improve behavior; curated examples work better.
2. **Missing eval loops:** No automated testing → failures discovered in production → expensive fixes.
3. **No failure budget:** Agent loops infinitely instead of failing gracefully after 3–5 attempts.
4. **Context rot:** As agent context window fills, model accuracy degrades (attention budget exhaustion).
5. **Memory poisoning:** Agent learns bad behavior from early mistakes, then repeats those mistakes later.

**Concrete symptom:**
- Agent's performance degrades after it's been running for 8 hours (context rot).
- Agent repeats the same mistake even after one correction (memory poisoning).
- Agent loops forever trying to fix one task (no iteration budget).
- First 100 tasks work great; tasks 101–200 fail mysteriously (no regression evals).

**Real data (2026):**
Teams without evals get stuck in reactive loops: fix bug 1 → creates bug 2 → fix bug 2 → creates bug 3. Teams with evals break this cycle—failures become test cases, tests prevent regressions, metrics replace guesswork.

**How it manifests:**
- Production systems that worked in testing fail silently.
- Expensive compute costs with no visibility.
- "Why does it work sometimes and fail other times?" (debugging nightmare).

## Part 2: Best Practices Catalog

### Best Practice #1: Sharpen the Axe—Invest in Upfront Setup

**Maps to pitfalls:** #6 (prompt-and-pray), #3 (token inefficiency), #5 (tools without direction)

**What it means:**
Before scaling an AI system, invest in:
- **Clear, written specification** of what done looks like.
- **Tool design:** Minimal set, clear purpose, unambiguous inputs.
- **Curated examples:** 3–5 diverse, representative examples trump 50 edge cases.
- **Context optimization:** Ruthlessly cut tokens that don't add signal.

**Why it works:**
- Models perform better with tight, high-signal context than with bloated context.
- Clear specs reduce revision cycles.
- Good tool design prevents wrong-tool calls.

**Concrete mining example:**
Instead of: "Use AI to schedule shifts" (vague)
Write: "Given current equipment status, personnel availability, and safety constraints, generate a 24-hour shift schedule that maximizes OEE (Overall Equipment Effectiveness) while keeping all safety metrics in green. Constraints: no person works >12 hrs/day, equipment X requires 2-person crew, night shifts pay 1.25x."

**Implementation checklist:**
- [ ] Write a 1–2 page spec (what success looks like).
- [ ] Define test cases (3 examples the AI should handle, 3 edge cases it should reject).
- [ ] Audit context: remove any token that doesn't increase model accuracy.
- [ ] Document tool boundaries: when to use tool A vs. tool B.

**Cost/benefit:**
- Upfront: 2–4 hours of specification.
- Saves: 10–20 hours of rework, 40–60% in wasted compute tokens.

---

### Best Practice #2: Spec-Driven AI Work (Write the Spec First)

**Maps to pitfalls:** #6 (prompt-and-pray), #1 (AI slop), #2 (hallucination/bias)

**What it means:**
The specification is the primary artifact. The AI generates code/content/analysis *against* the spec, not instead of it.

Three levels:
1. **Spec-first:** Spec written by humans, used to drive AI generation.
2. **Spec-anchored:** Spec stays updated after implementation (living document).
3. **Spec-as-source:** The spec itself is the primary source; code/content is derived from it.

**Why it works:**
- Shared architectural understanding across teams and across multiple AI agents.
- Reduces hallucination: AI is grounded to explicit requirements, not guessing.
- Spec drift is visible: if spec and output diverge, you catch it early.
- Reduces AI slop: AI output is measured against spec, not against "sounds good."

**Concrete mining example:**
**Spec:** "Equipment Maintenance Alert System"
- Trigger: Vibration exceeds 3.5 mm/s for >5 consecutive minutes.
- Action: Send alert to maintenance lead via SMS + Slack.
- Format: "[ALERT] Excavator-12: Vibration at 4.2 mm/s. Recommended: check tooth wear."
- Data source: Only real-time vibration sensors (not historical guesses).

**Implementation checklist:**
- [ ] Write spec in structured format (requirements, constraints, examples).
- [ ] Version spec in git (track changes).
- [ ] Map each requirement to eval test cases.
- [ ] Run AI-generated output against spec (automated checks).

**Cost/benefit:**
- Upfront: 4–6 hours per feature.
- Benefit: AI output is measurable, revisions drop 50–70%.

### Best Practice #3: Long-Running Agents (vs. Fragile One-Shots)

**Maps to pitfalls:** #4 (correction loops), #7 (agentic anti-patterns)

**What it means:**
Don't treat each AI call as isolated. Instead, design systems where the AI:
- Maintains internal state (what did I already try?).
- Self-corrects mid-task without restarting.
- Learns from mistakes within a single run.
- Gracefully fails if it hits limits (iteration budget).

**Why it works:**
- One-shot prompts fail on messy, real-world data.
- Long-running agents can say: "I tried Tool A, it failed, let me try Tool B instead."
- Agents that track their own history avoid repeating the same mistake 5 times.

**Concrete mining example:**
**One-shot (bad):** "Generate a daily production report."
→ AI generates something generic, no adaptation if data is missing.

**Long-running (good):**
1. Agent attempts to fetch daily production data from primary source.
2. If unavailable, tries backup sensor.
3. If backup is also unavailable, marks data as incomplete + alerts operator.
4. Composes report with whatever data is available + confidence flags.
5. Records what worked/didn't work for the next run.

**Implementation checklist:**
- [ ] Design agent with explicit loop: plan → execute → check → adapt.
- [ ] Set iteration limit (e.g., max 5 attempts before failing gracefully).
- [ ] Implement state tracking: what did the agent try?
- [ ] Add self-correction prompts: "Did that work? If not, what's your next step?"

**Cost/benefit:**
- Upfront: 1–2 days of system design.
- Benefit: Handles 80% of real-world messiness without user intervention.

### Best Practice #4: Plugins / Skills for Reusability

**Maps to pitfalls:** #5 (tools without direction), #3 (token inefficiency)

**What it means:**
Instead of a monolithic AI system, build **modular plugins** that AI agents can call:
- Each plugin has a single, clear purpose.
- Plugins are versioned and tested independently.
- Agents compose plugins to solve larger problems.

**Why it works:**
- Smaller, testable units reduce bugs.
- Reusable across multiple agents and workflows.
- Easy to version: if plugin v1 works, keep using it; upgrade to v2 when ready.
- Reduces token overhead: agents don't need the full spec of every tool, just the one they're using.

**Concrete mining example:**
Instead of: One giant "mining AI" that knows about equipment, schedules, safety, and compliance.

Use modular plugins:
- `equipment_status_plugin`: Returns real-time equipment state.
- `shift_scheduler_plugin`: Given constraints, generates shifts.
- `safety_checker_plugin`: Validates decisions against safety rules.
- `compliance_plugin`: Checks mining regulations.

Agents compose these: "Check equipment status → validate safety → generate shift → verify compliance."

**Implementation checklist:**
- [ ] Identify domain boundaries (equipment, scheduling, safety, etc.).
- [ ] Create one plugin per boundary with clear API.
- [ ] Write plugin spec (inputs, outputs, constraints).
- [ ] Test plugin independently before agents use it.
- [ ] Version plugins; allow agents to specify version.

**Cost/benefit:**
- Upfront: 1 week to design plugin architecture.
- Benefit: 50–70% reduction in per-task token cost; easier to debug; reusable across 5+ workflows.

### Best Practice #5: Context and Session Management

**Maps to pitfalls:** #3 (token inefficiency), #7 (agentic anti-patterns), #4 (correction loops)

**What it means:**
Be intentional about what information is *loaded into the AI's context* at each step:

**Levels:**
1. **Just-in-time retrieval:** Load only the data needed for the current step (not the entire database).
2. **Session compaction:** Summarize old conversation history instead of repeating it.
3. **Context prioritization:** High-signal info (specs, examples) loaded always; low-signal info loaded only when needed.
4. **Context rot mitigation:** Monitor model accuracy as context grows; trim if needed.

**Why it works:**
- Smaller context window = faster response + lower cost + better focus.
- Transformers have "attention budget": 100K tokens ≠ 100 times better than 1K tokens.
- Repeated context (copy-paste same background info) is wasted tokens.

**Concrete mining example:**
**Bad:** Every AI call loads the entire mining operations manual (500 pages).

**Good:**
- Query about equipment → load only equipment manual section (5 pages).
- Query about schedules → load only scheduling constraints (2 pages).
- After 10 conversation turns → compress: "In the last 10 turns, we determined: equipment X is down until 6pm, team A is available 2–10pm, priority is production target Y."

**Implementation checklist:**
- [ ] Map each AI task to "what context is truly needed?"
- [ ] Implement just-in-time retrieval: fetch only what's asked for.
- [ ] For multi-turn conversations, summarize after 10 turns.
- [ ] Monitor: Does accuracy drop as context grows? If yes, trim.
- [ ] Set context limit: "Don't exceed 50K tokens" (or your org's threshold).

**Cost/benefit:**
- Upfront: 1–2 days to refactor context loading.
- Benefit: 30–50% token reduction, 20–30% faster responses.

### Best Practice #6: Eval-Driven Iteration

**Maps to pitfalls:** All of them. This is the meta-practice.

**What it means:**
Before deploying or scaling AI, run automated tests:
1. **Code-based graders:** "Does the output match the spec?" (objective, fast).
2. **Model-based graders:** "Does this feel right?" (using another AI to judge).
3. **Human graders:** "Does this actually work?" (expert review, spot-checks).

**Why it works:**
- Teams without evals get stuck fixing one bug, which creates another bug (reactive hell).
- Teams with evals ship faster and break less.
- Evals act as regression tests: if a new change breaks previous functionality, you catch it *before* shipping.

**Concrete mining example:**
**Eval test 1 (code-based):**
Input: "Equipment X failed at 3pm."
Expected output: Alert sent to maintenance lead within 60 seconds.
Test: Did the system call the alert API?

**Eval test 2 (model-based):**
Input: Equipment status report from AI.
Grader prompt: "Is this report accurate and specific to Berau Coal's equipment?"
Threshold: Pass if 4/5 judges say "accurate."

**Eval test 3 (human-based):**
Pick 20 random system outputs per month.
Expert review: 100% accuracy required for production alerts.

**Implementation checklist:**
- [ ] Define 3–5 eval tests (one per major task).
- [ ] Set passing threshold (e.g., 95% on code-based, 90% on human).
- [ ] Run evals in CI/CD: every change is tested before merge.
- [ ] Track metrics over time: Are we improving? Regressing?
- [ ] Schedule monthly human review: spot-check 20 outputs.

**Cost/benefit:**
- Upfront: 2–3 days to design evals.
- Ongoing: 1–2 hours/week to maintain + run evals.
- Benefit: 80% fewer production bugs, confidence to scale, measurable improvement.

## Part 3: Mindset Shift—From Fear to Amplification

### The Fear (Real and Legitimate)

**Fear statement:** "AI will replace me. My job will disappear. I should resist this or it's too late."

**Why it's legitimate:**
- Real: 55,000 US job cuts in 2025 explicitly attributed to AI.
- Real: 23% of employees at AI-adopting organizations fear job elimination within 5 years.
- Real: 120 million workers globally are "at medium-term redundancy risk" due to lack of reskilling.

### The Evidence

But the fuller picture:
- World Economic Forum projects: 92M jobs displaced BUT **170M new jobs created** by 2030 (net +78M).
- Companies investing in reskilling report **50% faster innovation**, **56% wage premium** for AI-skilled workers.
- Human + AI outperforms either alone: Freestyle chess champions (humans + AI) beat grandmasters and supercomputers in head-to-head matches.

### The Reframe: AI Amplifies You

**New mindset:** "The person who learns to wield AI will outpace the person who doesn't."

**Why it works:**
- AI is 1000x better at certain tasks (searching, writing templates, analyzing data).
- Humans are 1000x better at certain tasks (judgment, creativity, relationships, strategy).
- Human + AI > either alone.

**Concrete reframing:**

| Old Fear | New Reality |
|----------|-------------|
| "AI will write my reports." | "AI will draft 80% of my report; I spend 20% time reviewing, editing, adding judgment." |
| "AI will replace the equipment team." | "Routine diagnostics are automated; the team focuses on complex failures and preventive strategy." |
| "I'll be out of a job." | "The people doing my job today without AI upskilling will be out of a job; I'll be managing AI systems." |
| "Technology moves too fast." | "The people who learn 1–2 AI tools per year stay relevant; those who learn zero fall behind." |

### Quotable Reframings for Leadership

Use these in the workshop:

1. **"AI amplifies your best thinking, not your mistakes."**
   - If you're precise, AI is precise. If you're sloppy, AI is sloppy.
   - Garbage in → garbage out. But *good* input → *force-multiplied* output.

2. **"The job isn't going away; it's evolving."**
   - Equipment technician today: diagnose failures by hand, write reports by hand.
   - Equipment technician tomorrow: AI diagnoses 90% of failures automatically; technician handles the 10% that are weird, ensures AI reasoning is sound, improves procedures.
   - Same role, higher impact.

3. **"Competitors aren't worried about AI; they're worried about *your* AI skills."**
   - 56% wage premium for workers with AI skills (McKinsey, 2026).
   - Early adopters who learn this now will be the mentors/leaders in 5 years.

4. **"Learning AI is faster than you think."**
   - Not everyone needs to be an AI researcher.
   - Teams that allocate 2–3 hours/week to experimenting with AI tools see measurable ROI in 8 weeks.
   - The learning curve is steeper at first, then flattens.

5. **"AI is a team sport; solo AI + human beats any AI alone."**
   - You bring domain expertise (mining operations, safety, equipment).
   - AI brings pattern-matching at scale.
   - Together: new possibilities that neither could achieve alone.

### Deployment Strategy for This Mindset

**In the workshop:**
1. **Acknowledge the fear** (don't dismiss it): "It's natural to worry about being replaced."
2. **Show the data**: 170M new jobs created; 56% wage premium.
3. **Show the math**: Humans + AI > either alone (chess example, mining examples).
4. **Make it concrete**: "Here are 3 tasks your team will stop doing manually, and 5 higher-value tasks you'll do instead."
5. **Commit to reskilling**: "If you invest 2 hours/week, here's what you'll be able to do in 8 weeks."

**In the lab:**
- Pair AI newcomers with someone who's experimented.
- Celebrate small wins: "You just automated a 30-minute task."
- Collect feedback: "What do *you* think the next high-impact task is?"

## End Notes

This document is **concrete and actionable**, designed for mining professionals with zero AI background. Every pitfall has a real cost (money, time, safety). Every best practice has a concrete implementation path.

**Key takeaway for BCE leadership:**
- Bad AI = waste (reputation, money, people's time).
- Good AI = force multiplier (faster decisions, higher quality, people freed to do judgment work).
- The difference is **discipline, planning, and measurement**—not magic.

**For the lab:** Have participants identify one high-impact task in their workflow and work through:
1. **Spec:** What does "done" look like?
2. **Plugins:** What tools would this AI need?
3. **Evals:** How would we test it?
4. **Context:** What info is truly needed?

This turns abstract knowledge into muscle memory.
